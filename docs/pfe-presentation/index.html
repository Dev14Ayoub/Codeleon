1. Introduction

1.1 Contexte du projet

Le présent rapport fait état de l'avancement du projet de fin d'études Codeleon, une plateforme web destinée à l'apprentissage et à l'enseignement de la programmation. Le projet est mené dans le cadre de la Licence Professionnelle et se déroule sur une période d'environ trois mois. Ce document couvre la phase de développement la plus récente et précède la rédaction du mémoire final.

L'idée de départ est partie d'un constat fait pendant les travaux pratiques de programmation : une part importante du temps de cours est consacrée à l'installation des outils (JDK, IDE, Git, etc.) et à la résolution de problèmes de configuration sur les machines des étudiants, plutôt qu'à l'apprentissage des concepts eux-mêmes. Codeleon tente d'offrir un environnement de travail prêt à l'emploi, accessible depuis un simple navigateur.

1.2 Problématique

Plusieurs plateformes en ligne existent déjà pour écrire du code collaborativement (Replit, CodeSandbox, etc.), mais elles présentent toutes au moins un des inconvénients suivants : elles sont propriétaires, payantes au-delà d'un certain usage, et surtout, elles envoient le code des utilisateurs vers des serveurs externes. Pour un établissement universitaire, cette dépendance est souvent inacceptable.

La problématique abordée par Codeleon peut donc se formuler ainsi : comment proposer un environnement de programmation collaboratif, doté d'un assistant intelligent, qui puisse être hébergé entièrement sur un poste de travail standard sans GPU, sans aucune connexion à un service distant ?

1.3 Objectifs

  • Mettre en place un éditeur de code en temps réel, accessible depuis un navigateur, supportant plusieurs langages et la collaboration entre utilisateurs.
  • Permettre l'exécution sécurisée de code dans un environnement isolé.
  • Intégrer un assistant intelligent local capable d'aider l'utilisateur dans son travail, sans recourir à des services cloud.
  • Concevoir l'ensemble de manière à pouvoir tourner sur la machine de référence du projet : un ordinateur portable doté de 8 Go de mémoire et sans carte graphique dédiée.

2. Architecture du système

2.1 Vue d'ensemble

L'application est organisée en trois couches : une interface web côté navigateur, un serveur applicatif, et un ensemble de services tournant dans des conteneurs Docker (base de données, cache, base vectorielle, runtime d'IA). Le navigateur ne communique jamais directement avec les conteneurs : seul le serveur applicatif joue le rôle d'intermédiaire et applique les contrôles d'accès.

Le moteur d'exécution de code (qui lance le code Python soumis par l'utilisateur) est volontairement isolé dans son propre conteneur, sans accès réseau, et avec des limites de mémoire et de temps. Ce choix de sécurité est détaillé en section 3.4.

Figure 2.1 — Vue d'ensemble de l'architecture en trois couches.

L'ensemble du système est orchestré par Docker Compose. Une commande unique (start.ps1) suffit à lever la totalité de l'environnement : conteneurs, backend et frontend.

3. Fonctionnalités réalisées

3.1 Authentification

L'application propose deux mécanismes d'authentification : par mot de passe classique (avec hachage bcrypt), et via OAuth2 auprès de GitHub ou Google. Les jetons d'accès sont émis sous forme de JWT signés (HS512) avec une durée de vie de 15 minutes ; un jeton de rafraîchissement plus long est stocké dans Redis. Un rôle ADMIN permet d'accéder à une interface d'administration listant les utilisateurs, les projets, et quelques statistiques globales.

3.2 Tableau de bord

Après authentification, l'utilisateur arrive sur un tableau de bord récapitulant ses projets. Au fil du développement, plusieurs incréments ont été ajoutés à cet écran : compteurs de fichiers et de membres sur chaque carte de projet, recherche et tri côté client, possibilité d'épingler ou d'archiver un projet, création depuis un modèle (template) et flux d'activité récente affiché en barre latérale.

Figure 3.1 — Tableau de bord après authentification.
6

Quatre incréments ont été livrés pour cet écran durant la phase actuelle :

  • Cartes enrichies. Chaque carte expose le nombre de fichiers, le nombre de collaborateurs, la date de dernière modification, et un bouton pour copier le code d'invitation. La recherche et le tri (par date, par nom, par nombre de fichiers) sont effectués côté client à partir des données chargées.
  • Épinglage et archivage. Chaque utilisateur peut épingler ses projets favoris (qui remontent en tête de liste) ; le propriétaire d'un projet peut l'archiver, ce qui le retire de l'affichage par défaut sans le supprimer. Deux nouvelles tables (room_pins et une colonne archived_at) ont été ajoutées au schéma.
  • Templates de projets. Un catalogue de modèles est fourni avec l'application (Python vide, Java vide, projet React, terrain de jeu algorithmique). À la création d'un projet, l'utilisateur peut choisir un modèle ; le backend matérialise alors automatiquement les fichiers correspondants.
  • Flux d'activité. Une table room_events de type « journal d'événements » enregistre les actions significatives (création de fichier, exécution de code, question posée à l'IA, etc.). Une barre latérale dans le tableau de bord affiche les derniers événements toutes les 30 secondes.

Une décision technique mérite d'être mentionnée : pour afficher « dernière modification par X » sur chaque carte sans surcharger les requêtes de listing, une colonne last_edited_by_id a été ajoutée directement à la table des projets et mise à jour dans la même transaction que l'événement correspondant. Le coût est négligeable et le listing reste rapide même avec beaucoup de projets.

Endpoints REST associés

Pour le tableau de bord seul : GET /rooms, GET /rooms?archived=true, POST /rooms, GET /rooms/public, POST et DELETE /rooms/:id/pin, POST et DELETE /rooms/:id/archive, GET /events, et GET /templates.

3.3 Éditeur collaboratif

Le cœur de l'application est la room, c'est-à-dire l'écran où les utilisateurs éditent réellement le code. L'écran est divisé en trois panneaux : l'explorateur de fichiers à gauche, l'éditeur Monaco au centre, et l'assistant IA à droite. Plusieurs utilisateurs peuvent ouvrir la même room et éditer simultanément les mêmes fichiers ; Y.js fusionne leurs modifications grâce à un algorithme CRDT, ce qui garantit la convergence finale quelles que soient les latences réseau.

Figure 3.2 — Vue d'une room avec l'éditeur de code, l'explorateur de fichiers et l'assistant IA.

Une fonctionnalité d'import depuis GitHub permet à l'utilisateur de coller l'URL d'un dépôt public ; les fichiers texte sont automatiquement importés dans la room. Cela permet par exemple à un enseignant de préparer un TP sur GitHub, puis de le charger dans Codeleon pour le travailler en classe.

3.4 Exécution sandboxée

Lorsque l'utilisateur appuie sur le bouton Run, le code Python du fichier actif est envoyé au backend, qui lance un conteneur Docker éphémère (image python:3.12-slim) avec les paramètres de sécurité suivants :

  • --network=none : aucune connexion réseau possible depuis le conteneur ;
  • --memory=256m : limite stricte de mémoire vive ;
  • --cpus=0.5 : limite de temps processeur ;
  • un délai d'attente de 8 secondes, au-delà duquel le conteneur est tué.

Le code est transmis au conteneur via son entrée standard, et la sortie standard ainsi que la sortie d'erreur sont remontées au navigateur dans le panneau Output. À la fin de l'exécution, le conteneur est détruit ; aucun état n'est conservé entre deux exécutions.

3.5 Assistant IA local

Le panneau de droite contient un assistant intelligent, accessible par chat, qui peut répondre à des questions sur le code en cours d'écriture. L'ensemble du traitement se fait localement, sur la machine, sans appel à un service externe.

Le fonctionnement repose sur une technique appelée Retrieval-Augmented Generation (RAG), qui se décompose en deux phases :

  • Indexation. Les fichiers du projet sont automatiquement découpés en morceaux de quelques centaines de caractères, puis transformés en vecteurs numériques par un modèle d'embeddings (nomic-embed-text). Ces vecteurs sont stockés dans la base Qdrant.
  • Interrogation. Lorsque l'utilisateur pose une question, celle-ci est elle aussi transformée en vecteur, puis comparée aux vecteurs stockés pour récupérer les morceaux de code les plus pertinents. Ces extraits sont ensuite intégrés dans le prompt envoyé au modèle de chat (qwen2.5-coder:1.5b), qui génère une réponse en flux continu (Server-Sent Events) vers le navigateur.

En complément de cette mécanique de recherche, l'assistant reçoit également deux informations envoyées directement par le navigateur à chaque question : le contenu du fichier actuellement ouvert (limité à 16 ko côté serveur), et le message d'erreur de la dernière exécution si elle a échoué. Ces deux ajouts sont décrits en section 4.

Les conversations sont persistées dans la base de données. Chaque utilisateur retrouve son historique de discussion à la prochaine connexion. Le propriétaire d'une room dispose d'une vue supplémentaire qui lui permet de consulter, en lecture seule, l'historique de chaque membre invité (utile dans un contexte pédagogique). Une mention explicite est affichée aux invités pour les informer de cette possibilité.

Choix du modèle

Le modèle qwen2.5-coder:1.5b a été retenu après expérimentation. La version 0,5 milliard de paramètres, initialement utilisée, donnait des réponses peu cohérentes sur des tâches de code ; la version 3 milliards consomme trop de mémoire vive sur la machine cible. La taille intermédiaire offre le meilleur compromis dans le contexte donné.

4. Travail réalisé durant cette phase

Cette section présente les principaux apports de la phase actuelle de développement. Le détail technique de chaque amélioration figure dans les messages de commit (voir annexe A).

4.1 Enrichissement du tableau de bord

Les quatre incréments du tableau de bord décrits en section 3.2 ont été développés et livrés successivement, chacun avec sa propre série de tests. Cette progression incrémentale a permis de vérifier le bon fonctionnement de chaque ajout avant de passer au suivant.

4.2 Refonte de l'assistant IA

Initialement, l'assistant IA ne pouvait répondre que sur la base des fichiers explicitement indexés par l'utilisateur (qui devait cliquer sur un bouton Index). Cette approche présentait plusieurs limites :

  • L'indexation était manuelle : l'utilisateur devait penser à la déclencher après chaque modification de fichier.
  • Seul le fichier actuellement ouvert était indexé, et ce sous un nom de chemin codé en dur.
  • L'IA n'avait pas accès à la sortie d'erreur de la dernière exécution, ce qui rendait difficile la résolution des bugs « en direct ».

La refonte s'est faite en quatre étapes :

ÉtapeDescription
ModèlePassage du modèle qwen2.5-coder:0.5b à qwen2.5-coder:1.5b, et ajustement des paramètres de gestion mémoire d'Ollama.
Contexte directAjout dans la requête de chat de trois champs optionnels : le chemin du fichier ouvert, son contenu, et le contenu de la dernière erreur d'exécution. Ces informations sont intégrées au prompt système.
Indexation autoCréation d'un endpoint POST /index/all qui indexe en une seule requête tous les fichiers du projet. Le frontend déclenche cette indexation automatiquement avant chaque question, mais seulement si le contenu du projet a changé depuis la dernière indexation.
MémoireAjout d'une table room_chat_messages avec deux niveaux d'accès : un membre invité voit uniquement ses propres conversations ; le propriétaire de la room peut, via un sélecteur, consulter en lecture seule la conversation de chacun.

L'objectif global de cette refonte est résumé par le constat suivant : un système RAG sur un petit nombre de fichiers apporte relativement peu, car la quantité d'informations à filtrer est de toute façon réduite. Le bénéfice véritable apparaît lorsqu'on importe un dépôt GitHub plus volumineux. Pour le cas courant (rooms à quelques fichiers), l'injection directe du fichier ouvert et de l'erreur d'exécution est plus efficace que la recherche vectorielle.

4.3 Durcissement des outils de développement

Le script PowerShell start.ps1, qui démarre l'ensemble du stack en une commande, a été rendu plus robuste à plusieurs égards :

  • Détection effective d'une panne du démon Docker (la version précédente affichait à tort « Docker is up » même lorsque le démon était inaccessible, en raison d'une particularité de la gestion des erreurs natives par PowerShell).
  • Libération automatique des ports 8080 et 5173 au démarrage. Lorsque l'utilisateur ferme la fenêtre du backend sans passer par le script d'arrêt, un processus Java peut rester actif et bloquer le port ; un utilitaire Stop-PortListener parcourt désormais l'arbre des processus et libère le port avant tout nouveau lancement.
  • Démarrage du stack complet (avec les services IA) en une seule commande sans paramètre.

Le script d'arrêt stop.ps1 a bénéficié des mêmes améliorations. Ces ajustements ne sont pas spectaculaires, mais ils suppriment une source fréquente de frustration en cours de développement.

4.4 Corrections diverses

Plusieurs petits problèmes d'interface ont également été résolus durant cette phase : retour visuel du bouton « Invite » après une copie, comportement de défilement de l'éditeur qui entraînait à tort les barres latérales, message d'erreur réseau qui s'affichait à tort après chaque réponse de l'IA, gestion du retour HTTP 401 au lieu d'une redirection HTML pour les requêtes API non authentifiées. Le détail de ces corrections figure dans l'historique des commits.

4.5 Mise sous test

L'ensemble des nouveaux développements est couvert par des tests unitaires et d'intégration JUnit. La suite passe actuellement avec 85 tests sur 85 réussis. Les modifications de schéma de base de données sont gérées par sept migrations Flyway versionnées, ce qui garantit la reproductibilité de l'état du schéma sur n'importe quelle machine.

5. Difficultés rencontrées et solutions adoptées

Le développement n'a pas été linéaire. Plusieurs problèmes techniques ont nécessité un travail d'analyse pour être résolus. Cette section en retient les plus représentatifs.

5.1 Bouton « Run » sans effet apparent

Lors du premier appel à l'exécution de code, le bouton Run ne semblait rien faire. L'image Docker python:3.12-slim n'ayant jamais été téléchargée sur la machine, le démon mettait plusieurs dizaines de secondes à la récupérer ; ce délai dépassait le timeout de 8 secondes prévu pour l'exécution.

Téléchargement préalable de l'image avec docker pull python:3.12-slim au moment de l'installation. À terme, le backend pourra effectuer ce préchargement automatiquement au démarrage.

5.2 Port 8080 occupé après une fermeture incorrecte

Fermer manuellement la fenêtre du backend laisse un processus Java orphelin qui continue d'écouter sur le port 8080. Le démarrage suivant échoue alors avec un message « Port 8080 was already in use ».

Ajout d'un utilitaire PowerShell qui identifie et termine les processus écoutant sur un port donné, en parcourant l'arbre des processus enfants. Cet utilitaire est invoqué automatiquement au démarrage des scripts.

5.3 Message d'erreur réseau parasite après chaque réponse de l'IA

À chaque fin de réponse de l'IA, un message rouge « network error » s'affichait dans le panneau de chat, alors même que la réponse arrivait complètement. Une capture brute du flux Server-Sent Events a permis de confirmer que le serveur clôturait proprement la connexion ; le navigateur Chrome lève cependant une exception lorsqu'il détecte cette fermeture, et cette exception était mal interprétée côté client.

Le client traite désormais l'événement done, envoyé par le serveur en fin de réponse, comme le signal autoritaire de succès. Toute exception levée après cet événement est ignorée. Une véritable panne réseau, sans réception du signal done, continue de remonter une erreur visible.

5.4 Défilement involontaire de la page entière

Lorsque le code d'un fichier dépassait la hauteur de l'écran, l'utilisateur faisait défiler la page entière (avec l'explorateur de fichiers et l'assistant IA) au lieu de ne défiler que dans l'éditeur. La cause était une mauvaise contrainte de hauteur dans la mise en page Flexbox.

Ajustement des règles CSS : la racine est désormais fixée à la hauteur de la fenêtre avec overflow caché, et seul l'éditeur Monaco peut défiler verticalement.

5.5 Refus argumenté d'inclure Maven / Spring Boot dans les templates

Une demande a été formulée d'ajouter des templates Maven et Spring Boot au catalogue de modèles de projet. L'analyse de faisabilité a montré que cela imposerait l'accès réseau depuis les conteneurs d'exécution (pour télécharger les dépendances Maven), ce qui contredit directement le principe d'isolation. De plus, Spring Boot est un serveur, non un script court : le modèle d'exécution actuel (lancer, récupérer la sortie, terminer) ne s'y prête pas.

Cette demande a été reportée à une phase ultérieure et documentée comme telle dans la feuille de route du projet. Les templates actuels restent compatibles avec le modèle d'exécution Python en place.

6. Tests et qualité du code

La qualité du code a été suivie tout au long du développement à travers plusieurs pratiques :

6.1 Tests automatisés

La suite de tests backend comporte actuellement 85 tests JUnit, tous au vert. Ces tests couvrent :

  • Les flux d'authentification (inscription, connexion, rafraîchissement de jeton, OAuth) ;
  • La gestion des projets (création, listage, épinglage, archivage, templates) ;
  • La gestion des fichiers (création, renommage, suppression, droits d'accès) ;
  • Le moteur d'exécution de code, avec des cas nominaux et des cas d'erreur ;
  • Le système d'indexation IA et le service de chat (en utilisant des mocks pour Ollama et Qdrant) ;
  • L'historique des conversations IA, avec une attention particulière au contrôle d'accès entre membres ;
  • Le tableau de bord d'administration.

Le frontend est, quant à lui, vérifié par le compilateur TypeScript en mode strict et par le build Vite avant chaque commit.

6.2 Gestion du schéma de base de données

Toutes les modifications de schéma passent par Flyway. Sept migrations ont été appliquées à ce jour :

  1. Schéma initial : utilisateurs, jetons, rooms, membres ;
  2. Fichiers de rooms ;
  3. Support multi-fichiers ;
  4. Champs OAuth sur les utilisateurs ;
  5. Épinglage et archivage des rooms ;
  6. Journal des événements de room ;
  7. Historique des conversations IA.

6.3 Convention de commit

Tous les commits suivent la convention Conventional Commits avec un préfixe (feat, fix, chore, docs) suivi d'un scope. Cela rend l'historique lisible et facilite la rédaction du mémoire final.

6.4 Choix architecturaux favorisant l'évolution

Le moteur d'exécution est conçu sous forme de table de dispatch : ajouter un nouveau langage (Java, C, C++, etc.) consiste à ajouter une entrée et l'image Docker correspondante, sans toucher au reste du système. Cette extensibilité ne sera pas exploitée dans le cadre du PFE mais reste une option ouverte.

7. Limites et perspectives

Cette section liste de façon volontaire les limites actuelles du projet, ainsi que les évolutions envisageables après la soutenance.

7.1 Limites présentes

Exécution restreinte au Python

Bien que l'éditeur supporte la coloration syntaxique pour plus de vingt-cinq langages, seul le code Python peut effectivement être exécuté à travers le bouton Run. Cette limite est volontaire : l'ajout d'autres langages (Java, C, C++) demanderait peu de travail technique, mais a été reporté pour se concentrer sur la stabilisation du noyau du produit.

Absence de support Maven, Gradle ou Spring Boot

Pour les raisons exposées en section 5.5, les projets de type Maven, Gradle ou Spring Boot ne sont pas pris en charge. Leur ajout nécessiterait de revoir le modèle de sécurité du moteur d'exécution (notamment l'isolation réseau) et la durée d'exécution autorisée.

Pas de terminal embarqué

L'application ne propose pas de terminal interactif intégré à la room (à la manière de VS Code ou Replit). L'ajout d'un tel terminal, à l'aide de la bibliothèque xterm.js et d'un proxy WebSocket vers un shell bash dans un conteneur, est techniquement réalisable mais demanderait environ une journée de travail supplémentaire.

Templates avec structure seulement

Les modèles de projet créent les fichiers attendus, mais ces fichiers sont vides. Pré-remplir leur contenu demanderait une mécanique additionnelle (soit côté serveur via un encodeur Y.Doc, soit côté client via une colonne de contenu d'amorçage appliquée à la première connexion WebSocket). Cette amélioration est documentée comme évolution éventuelle.

7.2 Perspectives

  • Phase suivante. Rédaction du mémoire final, préparation de la présentation orale et de la démonstration en direct.
  • Après la soutenance. Selon le temps disponible, les évolutions suivantes peuvent être envisagées : ajout d'un moteur d'exécution Java mono-fichier, intégration d'un terminal Bash dans la room, et pré-remplissage du contenu des templates.
  • Plus long terme. Une mise en production éducative dans le cadre d'un cours de l'établissement, qui permettrait d'évaluer l'usage réel par des étudiants et d'ajuster l'ergonomie en conséquence.

8. Conclusion

L'objectif initial du projet — proposer un environnement de programmation collaboratif, doté d'un assistant intelligent, et entièrement auto-hébergeable sur une machine standard — a été atteint dans ses grandes lignes. L'application est aujourd'hui fonctionnelle de bout en bout : un utilisateur peut s'inscrire, créer un projet, inviter un collaborateur, écrire et exécuter du code Python, et obtenir l'aide d'un assistant qui voit ses fichiers et ses erreurs.

Le travail de la phase actuelle a porté principalement sur le tableau de bord, l'assistant IA, et la robustesse des outils de développement. Ces évolutions ont été pensées et réalisées de manière incrémentale : chaque ajout a fait l'objet de tests, d'un commit séparé, et d'une vérification manuelle avant de passer au suivant. Cette discipline a permis de livrer un produit stable, avec un historique de versions clair et un schéma de base de données maîtrisé.

Plusieurs choix techniques méritent d'être soulignés rétrospectivement. La séparation du moteur d'exécution dans un conteneur sans réseau a permis de respecter une exigence forte de sécurité sans complexifier le reste du système. L'utilisation de Y.js pour la collaboration temps réel a évité de devoir développer un protocole de synchronisation maison. Enfin, le choix d'un modèle de langage local, quantifié pour tenir sur la machine cible, a permis de respecter la contrainte d'auto-hébergement, au prix d'une qualité de réponses inférieure à celle de modèles cloud, mais suffisante pour les cas d'usage envisagés.

Les prochaines semaines seront consacrées à la rédaction du mémoire final et à la préparation de la soutenance. Le code et la base de données actuels constituent une assise stable pour cette phase, et toutes les fonctionnalités présentées dans ce rapport sont disponibles pour la démonstration.

« Le projet a évolué selon une démarche d'ingénierie classique : construire, mesurer, observer une faiblesse, raffiner. Cette démarche, appliquée notamment à l'assistant IA, a permis de transformer un pipeline initialement peu utile en un outil réellement aidant pour les cas d'usage cibles. »
